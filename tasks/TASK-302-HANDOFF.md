## Recomendaciones para Backend/API

1. Centralizar mensajes user-facing por codigo de error.
   - Evitar exponer textos en ingles como `Invalid email or password.`

2. Mantener mensajes API tecnicos en ingles si son internos, pero mapearlos en Web a copy final en espanol.

3. Revisar `api/src/lib/notifier.js` para tildes y tono.
   - Actualmente los correos funcionan, pero usan `revision`, `invitacion`, `accion`, `telefono`, `direccion` sin tildes.

4. Agregar plantilla de `Acceso creado` solo si Product decide enviar ese correo.

5. No loguear links completos de invitacion con token.

## Recomendaciones para Web Dev

1. Hacer una pasada de copy en UI primero.
   - Cambiar `Email` a `Correo`.
   - Cambiar `Password` a `Contraseña`.
   - Normalizar tildes si se permite Unicode.

2. Cambiar labels de botones a accion real.
   - `Confirmar compra` -> `Registrar compra`.
   - `Confirmar canje` -> `Redimir puntos`.
   - `Confirmar uso` -> `Aplicar beneficio`.

3. Actualizar estados vacios antes de tocar layout.

4. Mantener terminos de glosario en navegacion, titulos, botones, errores y CSV.

5. Revisar CSV headers para que coincidan con UI:
   - `Metodo` -> `Metodo de pago`.
   - `Tipo` debe distinguir compra, redencion, membresia y beneficio aplicado.

## Riesgos para QA

- Cambios de copy pueden afectar pruebas que busquen texto exacto.
- Correos con Unicode deben validarse en ACS/email real si se agregan tildes.
- No cambiar asuntos de correo en produccion sin coordinar con Release si QA usa esos asuntos para evidencia.
